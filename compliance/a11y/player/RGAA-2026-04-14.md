# RGAA 4.1.2 Audit Report — @remotion/player

**Audit date:** 14.04.2026
**Standard:** RGAA 4.1.2 (maps to WCAG 2.1 AA / EN 301 549)
**Pages audited:**

- https://www.remotion.dev/player (marketing & demo page)
- https://www.remotion.dev/docs/player/buffer-state (docs)

**Test environment:** Chrome + NVDA
**Method:** Manual testing using browser developer tools to inspect the DOM and accessibility tree, as well as other tools: Axe DevTools, NVDA, Colour Contrast Analyser
**Reference:** https://accessibilite.numerique.gouv.fr/methode/criteres-et-tests/
**Auditor:** Victoria (external)
**Scope:** The `/player` marketing page and `/docs/player/buffer-state` were sampled as part of the remotion.dev RGAA 4.1.2 audit (14.04.2026). Only criteria where findings were recorded are listed below — criteria not listed were not evaluated against the Player pages specifically.

---

## Findings

### Theme 6: Links

| #   | Criterion                                                           | Status | Observations / Evidence                                                                                                                                                           |
| --- | ------------------------------------------------------------------- | ------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 6.2 | Are identical links with the same destination consistently labeled? | NC     | On `/player` there are several links with identical text ("Learn more", "Read docs") that lead to different destinations. Links leading to different destinations should differ in text or accessible name. |

### Theme 7: Scripts

| #   | Criterion                                                             | Status | Observations / Evidence                                                                                                                                 |
| --- | --------------------------------------------------------------------- | ------ | ------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 7.1 | Is each script, if necessary, compatible with assistive technologies? | NC     | On `/player`, under the "Demo" heading, interactive elements for color selection have no accessible role or name.                                       |
| 7.2 | Is each script controllable by keyboard and any pointing device?      | NC     | On `/player`, under the "Demo" heading, color-selection controls cannot be operated with the keyboard.                                                  |
| 7.5 | Are status messages correctly rendered by assistive technologies?     | NC     | On `/player`, under the "Demo" heading, when a color is selected the information in the nearby video changes but is not announced to screen readers. |

### Theme 9: Information Structure

| #   | Criterion                                  | Status | Observations / Evidence                                                                                                                                 |
| --- | ------------------------------------------ | ------ | ------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 9.1 | Is the information structured by headings? | NC     | On `/player`, texts "Powered by Remotion" and "Reference application" are visually styled as headings but marked up with `<strong>`. `<strong>` is for emphasis — use proper heading elements. |

### Theme 11: Forms

| #    | Criterion                          | Status | Observations / Evidence                                                                                                                                  |
| ---- | ---------------------------------- | ------ | -------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 11.1 | Does each form field have a label? | NC     | On `/player`, the input field under the "Demo" heading has no persistent `<label>` — placeholder-only. Provide a visible, persistent label element.     |

---

## Priority Remediation Plan

| Priority | RGAA criterion | Recommended action                                                                                                            |
| -------- | -------------- | ----------------------------------------------------------------------------------------------------------------------------- |
| Critical | 7.1, 7.2       | Make `/player` "Demo" color-selection controls keyboard-operable with proper `role`/`aria-label` (ideally native `<button>`). |
| High     | 7.5            | Announce `/player` "Demo" color-change updates to screen readers via `aria-live="polite"` on the affected region.             |
| High     | 11.1           | Associate the `/player` "Demo" input with a persistent `<label>`.                                                             |
| Medium   | 9.1            | Mark "Powered by Remotion" and "Reference application" as real headings instead of `<strong>`.                                |
| Medium   | 6.2            | Differentiate link text for repeated "Learn more" / "Read docs" links on `/player`.                                           |

---

## Declaration

On the sampled Player pages, the marketing and demo page fails multiple RGAA 4.1.2 criteria, primarily in Scripts (role/name, keyboard, status messages), Forms (label), Information Structure (misused `<strong>`), and Links (ambiguous duplicate link text). No findings were recorded for `/docs/player/buffer-state` within the sample scope.

Findings are extracted from the broader remotion.dev RGAA 4.1.2 audit (Victoria, 14.04.2026).
